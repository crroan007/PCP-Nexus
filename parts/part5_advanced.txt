## 12 Lead Document Deduplication v4.4
How PCP Nexus prevents duplicate petition filings.
When a petition is amended and filed more than once, the system automatically compares the Date Submitted values. Only the newest submission is kept -- the older version is discarded. This ensures only one row per unique lead document appears in the CSV and only the most current filing enters the merged PDF.

## 13 Safe CSV Handoff v4.5
How your daily CSV report is securely transferred every night.
Each night, PCP Nexus transfers the day's in-progress CSV to its final archive destination using a verified, four-step process:
1. CopyThe in-progress CSV is copied to the archive destination. v4.5 This destination is now the root PDF output folder, bypassing the legacy DONE folder.
2. VerifyA SHA-256 hash confirms the copy is completely identical to the source file. If the hashes don't match, the handoff is aborted and retried.
3. DeleteThe in-progress copy is deleted only after verification passes.
4. Roll ForwardA blank CSV with tomorrow's date is created, ensuring late-arriving emails don't mix with today's completed data.

## 15 Keyword Monitoring & Exceptions
How the system flags documents that need human review.
PCP Nexus scans the Accepted Comments field of each email for specific legal keywords. When a match is found, the document processes normally but is also flagged for manual review.

### What Happens When a Keyword Is Detected
1. Document Processes NormallyThe PDF is renamed/merged and delivered as usual -- processing is not blocked.
2. Email Routes to ExceptionsThe source email is moved to the Exceptions folder for visibility.

## 15a Phase 2 Packet Stabilization v4.5.1
Preventing race conditions with simultaneous emails.
When multiple emails for the same envelope arrive at the exact same moment, a 5-minute stabilization window ensures all parts are collected before the merge begins. Previously, the merge could start before all emails were packetized, causing false NO_LEAD exceptions. This fix runs within the existing 20-minute aggregation window with no change to overall processing speed.

## 15b Exception Email Routing v4.5.1
Failed packets now route to AI Exceptions.
Emails from failed packets (e.g., NO_LEAD timeout) are now automatically moved to the AI Exceptions folder instead of remaining in the input folder. This makes all failures visible to staff for manual review. The background sweep also correctly routes orphaned emails from failed packets to Exceptions rather than Completed.

## 15c Late-Arriving Document Fix v4.5.2
Ensures all documents in a multi-email envelope are captured.
When the system scans multiple emails for the same envelope, it processes them in batches. Previously, if the lead document (AX40) arrived in a later batch than the exhibits (AXPM, AXPE), the system would skip the AX40 entirely because the packet already existed in the database. This caused false NO_LEAD exceptions even though all emails were present. Now, PENDING packets allow late-arriving documents to be downloaded and added. The system checks which files are already present and only downloads new ones, resetting the stabilization timer to ensure a complete merge.

## 15d Dashboard Exception Detail v4.5.2
Exception items now show meaningful error details.
NO_LEAD exception items on the dashboard previously displayed "No Logs" with no guidance on how to resolve them. They now show the specific error description (which parts were received, which document was missing), recovery instructions (clear PCP-Packetized tag to reprocess), and case metadata (case number, case style, court) when available.
